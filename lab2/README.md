# Lab 2: Обучение модели классификации удовлетворённости клиентов Olist

## Описание проекта
В данной лабораторной работе решается задача бинарной классификации удовлетворённости клиентов бразильского e-commerce Olist.  
Целевая переменная `is_satisfied` принимает значение 1 (удовлетворён) или 0 (не удовлетворён).  
Данные получены в результате выполнения лабораторной работы №1 и хранятся в MinIO (S3-совместимое хранилище) в партиционированном формате Parquet.

Проект включает:
- **Базовое решение** (логистическая регрессия с балансировкой классов).
- **Три гипотезы по улучшению модели**:
  1. Нелинейные ансамблевые модели (Random Forest, XGBoost, LightGBM).
  2. Отбор признаков с помощью `SelectKBest` перед обучением логистической регрессии.
  3. Специализированные методы работы с дисбалансом классов (`class_weight='balanced'`, SMOTE) в комбинации с Random Forest.
- Логирование всех экспериментов в **MLflow** (параметры, метрики, графики, модели).
- Воспроизводимость и управление зависимостями через **Poetry**.

## Почему выбраны именно эти гипотезы и модели?

### 1. Выбор baseline
В качестве базового решения взята **логистическая регрессия с `class_weight='balanced'`**.  
- Это простая, интерпретируемая модель, которая служит точкой отсчёта.  
- Автоматическая балансировка классов позволяет учесть дисбаланс (77% положительных отзывов) без ручного взвешивания.  
- Если более сложные модели не дают существенного прироста, нет смысла усложнять систему.

### 2. Гипотеза 1: Нелинейные ансамбли
**Почему:**  
Логистическая регрессия строит линейную разделяющую поверхность. В реальных данных часто присутствуют нелинейные зависимости, которые ансамбли решающих деревьев способны улавливать.  
**Выбранные модели:**
- **Random Forest** – устойчивый к переобучению, хорошо работает «из коробки».
- **XGBoost** – мощный градиентный бустинг, часто показывает лучшее качество на табличных данных.
- **LightGBM** – ещё одна быстрая реализация бустинга, которая строит деревья по листьям (leaf-wise) и может достигать более высокой точности.

### 3. Гипотеза 2: Отбор признаков
**Почему:**  
В датасете после feature engineering (лаб. 1) может быть много слабо информативных или избыточных признаков.  
Удаление шума способно повысить обобщающую способность модели и снизить риск переобучения, особенно для линейных моделей.  
Используем **ANOVA F-test (f_classif)** для оценки значимости каждого признака независимо, а затем оставляем k лучших перед логистической регрессией.  
Это позволяет проверить, можно ли упростить модель без потери качества.

### 4. Гипотеза 3: Работа с дисбалансом классов
**Почему:**  
Исходный дисбаланс (77% Satisfied) может приводить к тому, что модель «игнорирует» минорный класс, показывая высокие accuracy за счёт большинства.  
Стандартные методы борьбы:
- `class_weight='balanced'` – штраф за ошибку обратно пропорционален частоте класса.
- **SMOTE** – синтетически создаёт новые примеры минорного класса, балансируя обучающую выборку.

Оба метода применены к Random Forest, потому что:
- Древовидные модели более гибкие, чем линейные, и могут лучше использовать сбалансированные данные.
- Хотим выяснить, улучшит ли специальная обработка дисбаланса качество по сравнению с обычными ансамблями из гипотезы 1.

## Установка и запуск

### 1. Клонирование и установка зависимостей
```bash
git clone <repo>
cd lab2
poetry install
```

### 2. Запуск MLflow Tracking Server (Docker)
```bash
docker compose up -d
```
Сервер будет доступен по адресу http://localhost:5000.

### 3. Конфигурация
Создайте файл config.py со следующими переменными (пример):

```text
MINIO_ENDPOINT=http://localhost:9000
MINIO_ACCESS_KEY=minioadmin
MINIO_SECRET_KEY=minioadmin
MINIO_BUCKET=data-lake
FEATURES_DIR=processed/final_dataset
MLFLOW_TRACKING_URI=http://localhost:5000
MLFLOW_EXPERIMENT_NAME=olist_satisfaction_v6
```
### 4. Запуск экспериментов
- Все эксперименты (базовое решение + три гипотезы):

```bash
poetry run python -m src.experiment
```
- Одна модель (например, логистическая регрессия):

```bash
poetry run python -m src.experiment --model logistic
```
Результаты (метрики, графики) сохранятся в MLflow и будут доступны в веб-интерфейсе

## Анализ результатов экспериментов
### Используемые метрики
При дисбалансе классов accuracy может вводить в заблуждение. Поэтому ключевыми метриками выбраны:
- F1-score – гармоническое среднее precision и recall, хорошо подходит для несбалансированных данных.
- ROC AUC – не зависит от порога классификации, показывает способность модели разделять классы.
- Дополнительно выводятся precision, recall и отдельно метрики для каждого класса.

### Базовое решение (Baseline)
**Модель**: Логистическая регрессия с class_weight='balanced'.

Тестировались значения C = 0.1, 1.0, 10.0.

Результаты (на тесте):
- Accuracy ≈ 0.7925
- Precision (Satisfied) ≈ 0.8812
- Recall (Satisfied) ≈ 0.8447
- F1-score ≈ 0.8625
- ROC AUC ≈ 0.7939

Балансировка позволила достичь recall для класса Unsatisfied ≈ 0.62, что неплохо для линейной модели.

### Гипотеза 1: Нелинейные ансамбли
Модели: Random Forest, XGBoost, LightGBM с варьированием количества деревьев и глубины/темпа обучения.

Лучший результат (XGBoost, n=300, lr=0.05):
- Accuracy = 0.8433
- Precision = 0.8538
- Recall = 0.9612
- F1-score = 0.9043
- ROC AUC = 0.8008

Общие наблюдения:
- Все ансамбли значительно превзошли baseline: прирост F1 составил ~4.2 п.п.
- LightGBM (F1≈0.9041) и Random Forest (F1≈0.9038) также показали отличные результаты.
- Однако recall для Unsatisfied упал до 0.39–0.45 (против 0.62 у baseline). Модели стали агрессивнее предсказывать большинство.
- Кривые валидации показали, что увеличение max_depth после 15–20 не даёт значимого прироста, зато увеличивает время обучения.

### Гипотеза 2: Отбор признаков
Модель: Логистическая регрессия после отбора k лучших признаков (f_classif).

Тестировались k = 20, 30, 50.

Лучший результат (k=30 и k=50):
- F1-score ≈ 0.9002
- ROC AUC ≈ 0.7916–0.7938

Вывод:
Отбор признаков дал серьёзный скачок относительно baseline (F1 вырос на ~3.8 п.п.), практически догнав ансамбли. Это свидетельствует о том, что в полном наборе из 62 признаков много избыточных или шумовых. Простая модель после фильтрации становится конкурентоспособной, оставаясь легко интерпретируемой.

### Гипотеза 3: Работа с дисбалансом классов
Модели: Random Forest с class_weight='balanced' и Random Forest + SMOTE.

Результаты:
- Balanced RF: F1 ≈ 0.8885, Recall Unsatisfied ≈ 0.56
- SMOTE RF: F1 ≈ 0.8872, Recall Unsatisfied ≈ 0.55

Анализ:
- Эти модели лучше распознают неудовлетворённых клиентов (recall для Unsatisfied 0.55–0.56), что может быть важно с бизнес-точки зрения.
- Общий F1 немного ниже, чем у ансамблей без специальной балансировки, зато модель менее смещена в сторону большинства.
- SMOTE не дал преимущества перед простым взвешиванием классов, возможно, из-за того, что Random Forest и так достаточно устойчив к дисбалансу.

## Общие выводы
1. Лучшая модель по F1-score – XGBoost (n_estimators=300, learning_rate=0.05) с F1 = 0.9043 и ROC AUC = 0.8008. Она превосходит baseline на 4.2 п.п.
2. Нелинейные ансамбли (гипотеза 1) дают наибольший прирост качества, однако склонны «забывать» минорный класс. Если бизнес-задача требует одинаково хорошо детектировать оба класса, стоит рассмотреть модели с балансировкой (гипотеза 3) или тюнинг порога классификации.
3. Отбор признаков (гипотеза 2) – простой и эффективный способ улучшить линейную модель. Логистическая регрессия на 30–50 признаках приближается по качеству к бустингу, но гораздо легче интерпретируется.
4. Специализированная работа с дисбалансом (гипотеза 3) может быть полезной, если важнее высокая полнота по минорному классу. В нашем случае общий F1 немного уступает «чистым» ансамблям, зато recall для Unsatisfied выше.
5. Все эксперименты полностью залогированы в MLflow, что обеспечивает воспроизводимость. Для каждого запуска сохранены:
  - гиперпараметры,
  - метрики на валидации и тесте,
  - артефакты (confusion matrix, ROC-кривая, важность признаков, кривая валидации),
  - сериализованная модель.
6. Благодаря архитектуре данных (партиционирование, индексный файл _order_ids.parquet) загрузка и обновление датасета не требуют повторного чтения всего объёма, что делает систему ресурсоэффективной.
